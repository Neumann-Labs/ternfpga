**Headline: `DSP48 = 0` for every module.** Vivado confirms the ternary "multiply"
is pure LUT sign-select + CARRY4 adder logic — the claim that frees all 90 DSP48
slices for the non-ternary glue (norm/RoPE/softmax/requant). The compute lanes
are tiny (<2.5% LUTs), so the 35T has ample room for parallel lanes + the memory
subsystem. This is the central architectural bet, now silicon-validated rather
than asserted.

**Fmax caveats (honesty).** These are **synthesis** estimates (pre-place-and-route)
for the **unpipelined v0**. The critical path is the K=8 ternary adder tree
(~13–14 CARRY4 levels): `ternary_gemv` WNS = −5.612 ns and `ternary_gemv_sparse`
WNS = −4.656 ns against the aggressive 4.0 ns (250 MHz) target → ~104 / ~116 MHz.
That already sits in the dossiers' ~100–150 MHz range; pipelining the adder tree
(a Phase-1 step) raises it, and real Fmax comes after place-and-route with real
constraints. BRAM is 0 because the result memory (`y_mem`) is small and the weight
memory (DDR3) is modeled externally — on-chip BRAM appears when the memory
subsystem is built.

**Pipelining (cycle 7).** `ternary_dot_pipe` registers the reduction into 3 stages
(sign-select → two half-sums → final sum), dropping the critical path from ~14 to
**4 logic levels** and lifting Fmax from ~104 MHz to **~280 MHz** (2.7×, WNS +0.424 ns
@4 ns) — for 3 cycles of latency, still **0 DSP**, bit-exact over 800 streamed dots.
It also *shrinks* (149 LUTs vs 233 combinational), since registers break the tree
into cheaper pieces. Folding this lane into the GEMV/gather engines is a follow-up.

**Dense packing (cycle 8).** `ternary_unpack5` decodes a base-3 byte — **5 ternary
weights = 1.6 bits/weight** (the log₂3 = 1.585-bit optimum, 20% tighter than the
2-bit codes, 5× tighter than INT8) — into 5 lane codes in **36 LUTs, 0 DSP, 0 CARRY**
(Vivado folds the ÷3/%3 chain into pure LUTs), combinational. This is the on-DDR3
weight layout that minimizes the bandwidth decode is bound by; verified bit-exact
over all 243 valid bytes.
